# オフライン差分計算 PoC(ゲーム Phase 0 / 最重要ロジック)

mvp_dev_plan.md Phase 0 の「オフライン差分計算PoC(イベントキュー方式:移動無制限+採掘2hバジェット)」。
CLAUDE.md が最重要ロジックと名指しする部分。**Unity 実装前の"手計算と突き合わせる基準実装"**であり、
そのまま C# へ移植する仕様として使う。

## ⚠ 2つの参照実装(モデル世代が違う)
ゲーム仕様が進化したため、この PoC には**2世代**の参照がある。用途で使い分ける。

| ファイル | モデル | 用途 |
|---|---|---|
| `offline_sim.py` / `test_offline_sim.py` | **旧**:kg連続採掘・**着艦即売却**・精錬FIFO・イベントキュー詳細 | 旧仕様の詳細基準(精錬・部分便凍結等の厳密な挙動) |
| `apply_offline_ref.py` / `test_apply_offline_ref.py` | **現行**:確率**個数**の期待値・**在庫格納(売らない)**・距離連動B・γレア度 | 現行 `FleetSimulator.ApplyOffline` の1対1ミラー=**今のゲームの真実源** |

現行モデルの変更点(旧との差):採掘は確率個数([[mining-probabilistic]])、収入は在庫へ格納し店で手動売却
([[sell-model-change]])、産出個数は惑星ごと固定予算B・距離連動([[yield-balance-model]])。
現行 `ApplyOffline` は期待値の簡易計算で、旧の「飛行中便を B 超過後に完了させる」細部は簡略化
(`trips = ⌊min(elapsed, 2h)/tripTime⌋`)。バジェット=最初の2hウォールクロックのみ採掘、は両世代共通。

## モデル(確定仕様の反映)
| 項目 | 仕様 | 出典 |
|---|---|---|
| 収入 | 着艦の瞬間に売却(持ち帰り換金のみ) | 企画書8章 |
| 船 | 自動ループ 飛ぶ→掘る→満杯→帰る→売る→再出発 | 企画書6章 |
| 移動時間 | 実距離 ÷ 全船統一速度 S(線形) | 企画書7章 |
| オフライン | **移動は無制限・採掘のみバジェット**(初期2h) | 企画書11章 |
| 売却価格 | **受取時点(復帰時)の当日価格で一括** | 企画書8章 |
| 精錬 | 鉱石→金属(回収率ρ・処理能力R・FIFOキュー) | 企画書8章 |

### バジェット挙動(要点)
- バジェットは**開始からのウォールクロック上限** B=min(budget, offline)。
  「5h放置→有効採掘2h+停止3h」に対応。
- 天体到着が `t >= B` → 採掘せず**その場で凍結**(このオフライン中は搬入なし)。
- 採掘中に B を跨ぐ → 満杯にならず凍結=**未搬入**(復帰後に継続)。
- 満杯後の**帰還・売却は移動扱い**なので B 超過でも成立する(便が"帰ってくる")。

## イベントキュー方式
`ARRIVE_BODY` / `ARRIVE_STATION` の2種を時刻順の優先度キューで再生。
`offline_sec` を超えるイベントは破棄(復帰後)。船ごとに独立に自動ループを継続。
精錬所は搬入イベント時に「前回更新からの経過×R」だけ消費して金属化(連続処理を離散イベントで表現)。

## テスト(手計算と一致・13件)
ベースライン/バジェット4分岐(新便停止・部分採掘凍結・帰還後売却・T<budget)/複数船独立/
初期状態(採掘中・帰還中)/精錬FIFOキュー/直売vs精錬トレードオフ/受取時点価格上書き/
**balanceデータ統合**(月片道1分・周期5分・2hで24便)。期待値は各テストにコメントで手計算。

## 既定の前提と【要検討】
- **精錬は採掘バジェットの対象外**(station側処理として継続)を既定とした。
  採掘バジェットに精錬も含めるかは未決定 → `refining_under_budget` フラグで切替可能に用意。
- 売却は受取時点価格で一括(便ごとの帰還時点価格は使わない=企画書の簡素化どおり)。
- 座標・角度・LOD は扱わない(移動時間だけが差分計算の本質)。

## C# 移植メモ
- `heapq` → `SortedSet`/`PriorityQueue<T>`。dataclass → struct/class。
- 浮動小数の丸めは Unity 側の通貨表示(K/M/B・明細)に合わせて集約時に実施。
- `simulate_offline` の入出力(Ship/Body/Refinery/OfflineResult)がそのまま
  復帰ダイアログ「N便帰還・+XXX NOVA」明細のデータ源になる。
