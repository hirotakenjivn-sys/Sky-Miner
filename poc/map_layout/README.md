# 対数リング配置 PoC(ゲーム Phase 0 / 付録A の座標系)

企画書 付録A の「放射状・対数距離リング+惑星系クラスター+千鳥角度」配置を、
実データ(`data/balance/bodies.json` の50天体)から計算する基準実装。
ズーム PoC(Unity で60fps測定)の**"どこに何を置くか"の数理を Unity 非依存で確定**し、
そのまま C# へ移植する。座標はワールド座標、ズーム(カメラ)は Unity 側。

## 実行
```bash
python3 poc/map_layout/ring_layout.py                    # 配置計算 + build/map_layout.svg 出力
python3 -m unittest poc.map_layout.test_ring_layout      # 付録A成立性チェック11件
open build/map_layout.svg                                # 目視確認(B7土星圏を展開表示)
```

## 配置ルール(付録A R1〜R6)と実装
| ルール | 実装 |
|---|---|
| R1 対数距離リング | バンドを距離順に並べ `r = 前リング + max(K×log10比, min_ΔR)`。実距離は別カラム保持 |
| R2 リング間隔の下限 | `ΔR ≥ アイコン径×2.5`(min_ΔR)。単調増加を保証 |
| R3 クラスター | 惑星系は親が1スロット、衛星は親周囲サブリング(半径 ≤ ΔR×0.4)。集約=セマンティックズーム |
| R4 角度配置 | 同一リング等角+隣接リング半ピッチ千鳥。重なりを数値検証 |
| R5 LOD | オーバービュー=クラスター集約後の28ノードのみ(衛星は親へ集約) |
| R6 縦画面 | ワールド座標。ビューポート内接スケールは Unity 側 |

## 検証結果(実データ50天体)
- 50天体配置 → **オーバービュー28ノード**(6クラスタで衛星22体を集約:火星2/木星4/土星7/天王星5/海王星3/冥王星1)
- リング B1..B10 = 120,229,289,349,409,469,529,589,649,709(px相当)
- **オーバービュー最小ペア間距離 60px ≥ アイコン径24 = 重なりゼロ**(水星×火星が最接近)
- 全リングで弧長 ≥ アイコン径(角度的に収まる)

## 設計上の発見(記録)
- **B3(内惑星)以降はリング間隔が min_ΔR に張り付く**:実距離の log 比が小さく、
  下限で押し広げられるため。これは付録Aが言う「10バンドがほぼ等間隔のリングに正規化」
  の狙いどおりの挙動。log の性格は主に **B1→B2 の大ジャンプ**(月→近傍小惑星、
  片道×13の"惑星間航行エンジン解禁"演出)に表れる。
- **B3水星 と B4火星 が同角度で最接近**(スロット数の関係で千鳥が効かない場合がある)。
  重なりは無い(ΔR=60)が、ラベル密集を避けるため LOD でラベル間引き(R5)を推奨。
- 衛星の多いバンド(B7土星=7衛星)はサブリング上で衛星同士がワールド座標では近接する。
  これは**ズームで分離**する前提(展開閾値=クラスター直径80px超, R3)。オーバービューでは
  集約1アイコンなので問題にならない。

## C# 移植メモ
- `layout()` の出力 `Placed`(no/band/type/ring_radius/angle/x/y/role/parent_no)が
  そのまま Unity のマップ生成データ。`role`(standalone/cluster_parent/moon)で描画とLODを分岐。
- `bodies.json` の `有効化フラグ` は未実装(MVPは B1〜B5 の5天体のみ有効化)。
  Unity 側で `no ∈ {0(月),1(エロス),6(水星),7(火星),10-15(小惑星帯)}` 等の有効化を重ねる。
- SVG エクスポートは Unity Gizmos での可視化に相当(移植不要のデバッグ補助)。
