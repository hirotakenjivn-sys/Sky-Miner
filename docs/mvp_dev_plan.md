# MVP開発計画書 v1(宇宙採掘ゲーム / iOS)

## 0. 前提
- 体制:開発1名(+AI支援でコード生成・アセット下書き)。Unity経験の深さで工数は±30%変動
- エンジン:Unity 6(2D URP)+ C#。対象 iOS 16+、縦画面、60fps目標(下限機種30fps)
- サーバ:別紙「市況連動サーバ設計書」(Python/SQLite、開発はゲームと独立に進行可)
- バランス:credit_economy_model.xlsx を単一の真実源とし、CSV書き出し→Unityへ取り込む表駆動設計
- 見積単位:人日(1日=集中4〜6時間換算)。**合計 約84人日 = フルタイムで約4ヶ月、週20時間なら7〜8ヶ月**

## 3. リスクと対策
| リスク | 影響 | 対策 |
|---|---|---|
| シームレスズームの実機性能 | 大 | Phase 0で最初に検証。不合格ならLOD段階遷移へ設計後退(企画書5章の判断を差し戻し) |
| オフライン計算の不整合(複数船×精錬キュー×2hバジェット) | 大 | イベントキュー方式に一本化し、ユニットテストをPhase 0から整備 |
| 価格データのライセンス | 中 | MVPはFRED系+手動入力のみで審査提出可能な構成にし、購読系はライセンス確定後に追加 |
| IAP審査リジェクト | 中 | リストア・価格表示・復元導線をAppleガイドライン通りに。ガチャなし・広告なし構成で初回提出 |
| バランス崩壊 | 中 | 全数値を表駆動+リモート更新可能に(価格サーバに balance.json を相乗り配信) |
| 1人開発の中断 | 中 | 各Phaseの完了条件を「遊べる状態」に置き、どこで止まっても動くビルドが残るようにする |

## 4. MVPスコープの最終確認(企画書14章と同期)
- 含む:5天体/2船種/シームレスズーム/持ち帰り換金/精錬(鉄・Ni・Ti)/オフライン2h/強化/GEM+IAP/チュートリアル/市況連動(FRED系+固定値)/単価常時表示
- 含まない:合金加工(v1.1)/納品契約/マーケット図鑑フル版/6天体目以降/リワード広告(v1.1で追加検討)
- 【スコープ判断枠】マーケットタブ簡易版(2人日):初回リリースの独自性訴求として推奨だが、遅延時は最初に切る
